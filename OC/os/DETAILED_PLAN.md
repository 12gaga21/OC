# 📜 ДЕТАЛЬНЫЙ ПЛАНЪ РАЗРАБОТКИ: «ПУТЬ КЪ ИМПЕРСКОМУ ПРЕВОСХОДСТВУ»

**Цѣль:** Превратить ОС «Священный Ритуалъ» въ полноценнаго конкурента Linux, сохранивъ уникальность (ассемблеръ, стиль, безопасность).
**Статусъ:** Активная фаза разработки (v0.2 -> v1.0)
**Обновлено:** 5 Апрѣля 2026 г.

---

## 🏛️ I. АРХИТЕКТУРНЫЕ ПРИНЦИПЫ (НЕПРЕЛОЖНЫЕ ЗАКОНЫ)

1.  **Чистый Ассемблеръ:** Никакого C/C++. Только NASM. Максимальная производительность и контроль.
2.  **SYSLIB (Единая Библіотека):** Всѣ программы используют только вызовы `INT 0x80`. Прямой доступъ къ железу запрещенъ Инквизиціей.
3.  **Безопасность прежде всего:** Модуль «Святая Инквизиція» проверяетъ каждую операцию памяти и доступъ къ файламъ.
4.  **Стиль «Злато-Изумрудный Ритуалъ»:**
    *   Цвѣта: Изумрудный текстъ (0x0A), Златыя рамки (0x1E), Черный фонъ (0x00).
    *   Орѳографія: Дореформенная (ѣ, і, ъ, ѳ).
    *   Тонъ: Высокій, ритуальный («Благословеніе», «Ересь», «Стражъ»).

## 🗺️ II. ДОРОЖНАЯ КАРТА (ПОЭТАПНО)

### ЭТАП 1: ФУНДАМЕНТЪ БЕЗОПАСНОСТИ (Недѣли 1-2)
*Цѣль: Реализовать изоляцию процессовъ и виртуальную память.*

#### Задача 1.1: Активация Paging (Страничная адресация)
*   **Файлъ:** `src/kernel/memory.asm`, `src/kernel/kernel.asm`
*   **Дѣйствія:**
    1.  Создать каталогъ страницъ (Page Directory) и таблицы страницъ (Page Tables) для ядра (первые 4MB).
    2.  Настроить регистръ `CR3` (адресъ каталога).
    3.  Включить битъ PG (Page Global) въ регистрѣ `CR0`.
    4.  Реализовать обработчикъ исключенія `#PF` (Page Fault, Interrupt 14) для ленивой подгрузки страницъ.
*   **Критерій успѣха:** Ядро работаетъ въ защищенномъ режимѣ съ виртуальной памятью. Ошибки доступа ловятся.

#### Задача 1.2: Изоляция пространствъ адресовъ
*   **Файлъ:** `src/kernel/loader.asm`, `src/kernel/process.asm`
*   **Дѣйствія:**
    1.  При загрузкѣ программы выдѣлять ей отдельныя страницы памяти.
    2.  Менять `CR3` при переключеніи задачъ (контекстный переходъ).
    3.  Помѣтить страницы ядра какъ Supervisor (битъ U/S = 0), страницы пользователя какъ User (битъ U/S = 1).
*   **Критерій успѣха:** Пользовательская программа не можетъ прочитать память ядра (вызываетъ #PF, который ловитъ Инквизиція).

#### Задача 1.3: Усиліе «Святой Инквизиціи»
*   **Файлъ:** `src/security/inquisition.asm`
*   **Дѣйствія:**
    1.  Внедрить проверку хэш-суммы (CRC32 или SHA-1 на ассемблерѣ) для запускаемыхъ `.bin` файловъ.
    2.  Вести «Черную Книгу» (логъ) нарушеній въ буферѣ VGA или COM-портъ.
    3.  Блокировать запускъ кода безъ подписи (магическое число въ заголовкѣ файла).

---

### ЭТАП 2: ФИЛОВАЯ СИСТЕМА «ЛѢТОПИСЬ» (Недѣли 3-4)
*Цѣль: Надежное храненіе данныхъ съ журналированіемъ.*

#### Задача 2.1: Модернизация FAT драйвера
*   **Файлъ:** `src/drivers/fs_fat.asm`, `src/kernel/fs_journal.asm`
*   **Дѣйствія:**
    1.  Интегрировать журналы транзакцій (Write-Ahead Logging) въ операціи записи.
    2.  Передъ записью въ секторъ писать «намѣреніе» въ журналъ.
    3.  При загрузкѣ проверять флагъ «грязный томъ» и воспроизводить журналъ (Replay).
*   **Критерій успѣха:** После сбоя питания (reset QEMU) файлы не повреждаются.

#### Задача 2.2: Права доступа (Аналогъ chmod/chown)
*   **Файлъ:** `src/kernel/fs_permissions.asm`
*   **Дѣйствія:**
    1.  Добавить метаданныя къ файламъ (владелецъ, группа, права: R/W/X).
    2.  Реализовать системные вызовы `sys_chmod`, `sys_chown`.
    3.  Проверять права передъ каждымъ `open/read/write`.
*   **Критерій успѣха:** Пользователь не можетъ удалить системный файлъ безъ правъ «Архіерея».

### ЭТАП 3: ЭКОСИСТЕМА И ПРИЛОЖЕНІЯ (Недѣли 5-6)
*Цѣль: Возможность запускать полезныя программы.*

#### Задача 3.1: Команда `run` въ Shell
*   **Файлъ:** `src/shell/shell.asm`
*   **Дѣйствія:**
    1.  Парсить аргументы командной строки.
    2.  Искать файлъ въ текущемъ каталогѣ и PATH.
    3.  Вызывать `loader_load_program` и передавать управленіе.
    4.  Возвращать кодъ завершения въ shell.

#### Задача 3.2: Наборъ утилитъ (Userland)
*   **Файлы:** `apps/` (новая папка)
*   **Списокъ программъ:**
    1.  `hello.bin`: Выводъ привѣтствія (проверка SYSLIB).
    2.  `calc.bin`: Простой калькуляторъ (аргументы изъ CLI).
    3.  `edit.bin`: Текстовый редакторъ (TUI окна).
    4.  `netcat.bin`: Отправка данныхъ по сети (проверка стека).
    5.  `info.bin`: Выводъ подробной информаціи о системѣ.

#### Задача 3.3: Скриптовый языкъ «Свитокъ»
*   **Файлъ:** `src/shell/script_parser.asm`
*   **Дѣйствія:**
    1.  Поддержка пакетной обработки (*.SCR файлы).
    2.  Переменныя `%VAR%`, условія `IF/ELSE`, циклы `FOR`.
    3.  Конвейеры `|` (передача вывода одной команды въ другую).

## 🛠️ III. ИНСТРУКЦІИ ДЛЯ ИИ-АГЕНТА (КАКЪ ПИСАТЬ КОДЪ)

1.  **Структура файла:**
    ```nasm
    ; † СВЯЩЕННЫЙ МОДУЛЬ: <ИМЯ>
    ; Назначеніе: <ОПИСАНИЕ>
    ; Стиль: Злато-Изумрудный
    ; Авторъ: Омниссія

    [bits 32]
    section .text
    global <function_name>

    <function_name>:
        ; Кодъ съ комментаріями на старославянскомъ
        ret
    ```

2.  **Комментаріи:** Обязательны для каждой функціи. Использовать слова: «Входъ», «Выходъ», «Разрушаетъ регистры», «Благословеніе».
3.  **Ошибки:** Использовать макросъ `PANIC <MSG>` для критическихъ ошибокъ. Выводить сообщенія въ красномъ цвѣтѣ.
4.  **Тестированіе:** Каждая новая функція должна быть протестирована въ эмуляторѣ QEMU передъ коммитомъ.

## ⚠️ V. РИСКИ И ПУТИ ПРЕДОТВРАЩЕНІЯ

1.  **Рискъ:** Сложность отладки ассемблера.
    *   *Рѣшеніе:* Активное использование логовъ черезъ COM-портъ и эмуляторъ GDB.
2.  **Рискъ:** Нехватка памяти (менее 4MB).
    *   *Рѣшеніе:* Оптимизація структуръ данныхъ, отказъ отъ избыточныхъ копій.
3.  **Рискъ:** Ошибки въ Paging (Triple Fault).
    *   *Рѣшеніе:* Поэтапное включеніе, тестированіе на каждой строкѣ кода.
