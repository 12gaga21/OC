# Внутреннее руководство ИИ-агента по созданию ОС на ассемблере

## 1. Внутренняя архитектура ИИ-агента

### 1.1. Основные компоненты
- **Анализатор запросов** (`src/modules/request_analyzer.py`): Обрабатывает входящие запросы от пользователя.
- **Планировщик задач** (`src/modules/task_planner.py`): Создает последовательность шагов для выполнения запроса.
- **Генератор кода** (`src/modules/code_generator.py`): Создает ассемблерный код для компонентов ОС.
- **Тестировщик** (`src/modules/tester.py`): Проверяет корректность сгенерированного кода.
- **Менеджер архитектур** (`src/modules/architecture_handler.py`): Адаптирует код под целевые архитектуры.
- **Интегратор Git** (`src/modules/git_integration.py`): Управляет версиями и фиксирует изменения.
- **Документатор** (`src/modules/documenter.py`): Создает документацию по сгенерированному коду.

### 1.2. Взаимодействие между компонентами
- Все компоненты взаимодействуют через центральный контроллер (`src/main.py`).
- Компоненты обмениваются данными через стандартизированные интерфейсы.
- Каждый компонент возвращает статус выполнения (успех/ошибка) и логи.

## 2. Процесс обработки запросов

### 2.1. Получение запроса
1. Принять запрос от пользователя через текстовый интерфейс.
2. Определить тип запроса (создание ОС, модификация компонента, отладка и т.д.).
3. Извлечь ключевые параметры (архитектура, функциональность, требования к ресурсам).

### 2.2. Анализ запроса
1. Проверить полноту информации в запросе.
2. Определить необходимые компоненты ОС.
3. Оценить сложность задачи и примерное время выполнения.
4. Проверить совместимость с поддерживаемыми архитектурами.

### 2.3. Планирование задач
1. Разбить запрос на конкретные подзадачи.
2. Определить последовательность выполнения подзадач.
3. Назначить каждую подзадачу соответствующему модулю.
4. Установить зависимости между подзадачами.

### 2.4. Выполнение задач
1. Последовательно выполнять подзадачи в соответствии с планом.
2. Для каждой подзадачи:
   - Вызвать соответствующий модуль
   - Передать необходимые параметры
   - Получить результат выполнения
   - Проверить корректность результата
3. При ошибках - попытаться исправить или запросить дополнительную информацию.

### 2.5. Тестирование и проверка
1. Проверить синтаксис сгенерированного ассемблерного кода.
2. Запустить эмуляцию для проверки функциональности.
3. Проверить соответствие требованиям из запроса.
4. Задокументировать результаты тестирования.

### 2.6. Предоставление результата
1. Подготовить отчет о выполнении запроса.
2. Сгенерировать документацию по созданным компонентам.
3. Зафиксировать изменения в системе контроля версий.
4. Предоставить пользователю ссылки на созданные файлы и документацию.

## 3. Спецификации модулей

### 3.1. Анализатор запросов
- **Вход**: Текстовый запрос от пользователя
- **Выход**: Структурированное описание задачи с параметрами
- **Функции**:
  - Извлечение ключевых слов и параметров
  - Определение типа задачи
  - Проверка полноты информации
  - Формирование технического задания

### 3.2. Планировщик задач
- **Вход**: Структурированное описание задачи
- **Выход**: Последовательность подзадач с зависимостями
- **Функции**:
  - Разбиение задачи на подзадачи
  - Определение последовательности выполнения
  - Назначение подзадач модулям
  - Управление зависимостями

### 3.3. Генератор кода
- **Вход**: Описание компонента ОС, целевая архитектура
- **Выход**: Ассемблерный код компонента
- **Функции**:
  - Генерация кода загрузчика
  - Генерация кода ядра
  - Создание драйверов устройств
  - Реализация файловой системы
  - Оптимизация сгенерированного кода

### 3.4. Тестировщик
- **Вход**: Ассемблерный код компонента
- **Выход**: Результаты тестирования, отчет об ошибках
- **Функции**:
  - Статическая проверка синтаксиса
  - Эмуляция выполнения
  - Анализ результатов
  - Формирование отчета

### 3.5. Менеджер архитектур
- **Вход**: Ассемблерный код, целевая архитектура
- **Выход**: Адаптированный код для целевой архитектуры
- **Функции**:
  - Определение различий между архитектурами
  - Адаптация кода под специфику архитектуры
  - Оптимизация под целевую платформу

## 4. Процесс принятия решений

### 4.1. Приоритезация задач
1. Определить критичность каждой подзадачи.
2. Учитывать зависимости между задачами.
3. Оптимизировать порядок выполнения для минимизации времени.

### 4.2. Выбор подходов
1. При наличии нескольких способов реализации - выбрать оптимальный.
2. Учитывать требования по производительности и ресурсам.
3. При неопределенности - запросить уточнение у пользователя.

### 4.3. Обработка ошибок
1. При ошибках в генерации кода - попытаться найти альтернативные подходы.
2. При критических ошибках - остановить выполнение и уведомить пользователя.
3. При незначительных ошибках - попытаться автоматически исправить.

## 5. Механизмы самоконтроля и проверки

### 5.1. Проверка корректности
- После каждой генерации кода выполнять синтаксическую проверку.
- Сравнивать сгенерированный код с эталонными шаблонами.
- Проверять соответствие требованиям из технического задания.

### 5.2. Мониторинг прогресса
- Отслеживать выполнение каждой подзадачи.
- Логировать промежуточные результаты.
- При отклонениях от плана - корректировать последовательность выполнения.

### 5.3. Самоанализ
- После завершения задачи анализировать эффективность выполнения.
- Определять возможные улучшения процесса.
- Обновлять шаблоны и подходы на основе полученного опыта.

## 6. Работа с ассемблером

### 6.1. Синтаксис и стандарты
- Строго следовать синтаксису целевого ассемблера (NASM, GAS).
- Использовать стандартные соглашения именования.
- Комментировать сложные участки кода.

### 6.2. Архитектурные особенности
- Учитывать различия в наборах инструкций.
- Использовать архитектурно-специфичные оптимизации.
- Правильно работать с регистрами и памятью.

### 6.3. Совместимость
- Обеспечивать совместимость с различными версиями ассемблеров.
- Проверять переносимость кода между эмуляторами.
- Следовать стандартам, применимым к целевой архитектуре.
