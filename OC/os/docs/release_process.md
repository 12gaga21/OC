# Процесс создания релизов

## Общие положения

Релизы операционной системы создаются автоматически при создании тегов в формате `v*` (например, `v1.0.0`, `v2.1.3`).

## Создание нового релиза

1. Убедитесь, что все изменения протестированы и готовы к релизу

2. Создайте новый тег в формате SemVer:
   ```bash
   git tag -a v1.0.0 -m "Первый стабильный релиз"
   git push origin v1.0.0
   ```

3. GitHub Actions автоматически запустит процесс создания релиза:
   - Сборка операционной системы
   - Создание артефактов (образ ОС, загрузчик, исходный код)
   - Генерация заметок к релизу
   - Публикация релиза на GitHub

## Формат тегов

Используется семантическое версионирование (SemVer) в формате `vMAJOR.MINOR.PATCH`:

- MAJOR: когда вы делаете несовместимые изменения API
- MINOR: когда вы добавляете функциональность обратно совместимым образом
- PATCH: когда вы делаете обратно совместимые исправления ошибок

Примеры:
- v1.0.0 - первый стабильный релиз
- v1.0.1 - исправление ошибок
- v1.1.0 - добавление новой функциональности
- v2.0.0 - несовместимые изменения

## Содержимое релиза

Каждый релиз включает следующие артефакты:

1. `os-image.img` - загрузочный образ операционной системы
2. `bootloader.bin` - отдельный загрузчик
3. `source-code.tar.gz` - архив с исходным кодом

## Автоматически генерируемые заметки к релизу

Заметки к релизу автоматически генерируются на основе коммитов между текущим и предыдущим релизами. Они включают:

- Список изменений
- Инструкции по использованию
- Известные проблемы (если есть)

## Ручное редактирование релиза

После автоматического создания релиза его можно отредактировать вручную через веб-интерфейс GitHub:

1. Перейдите на страницу релизов репозитория
2. Найдите нужный релиз
3. Нажмите "Edit release"
4. Внесите необходимые изменения
