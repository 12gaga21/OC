# Рекомендации по код-ревью для проекта ОС на ассемблере

## Общие принципы

1. Все pull requests автоматически проходят AI код-ревью через встроенный механизм SourceCraft
2. Ревьюеры должны обращать внимание на архитектурные аспекты и безопасность
3. Следует уделять особое внимание эффективности кода и работе с памятью

## Проверка стиля кодирования

### Именование
- Метки должны следовать консистентному стилю (например, snake_case)
- Имена функций должны отражать их назначение
- Глобальные переменные должны иметь префикс, указывающий на модуль

### Комментирование
- Все функции должны иметь комментарии с описанием назначения
- Сложные алгоритмы должны быть снабжены пошаговыми комментариями
- Регистры, используемые для передачи параметров, должны быть документированы

### Структура кода
- Соблюдение отступов (рекомендуется 2 пробела)
- Группировка связанных функций в одном файле
- Использование макросов для повторяющихся паттернов

## Архитектурные аспекты

### Работа с памятью
- Проверка корректности выравнивания данных
- Убедиться в правильности использования сегментов памяти
- Проверить освобождение временных буферов

### Обработка прерываний
- Убедиться в правильной установке обработчиков прерываний
- Проверить отправку сигнала EOI контроллеру прерываний
- Убедиться в сохранении и восстановлении регистров

### Ввод-вывод
- Проверить корректность работы с портами ввода-вывода
- Убедиться в правильной обработке ошибок при работе с устройствами

## Производительность

### Оптимизация
- Использование регистров вместо памяти для временных значений
- Минимизация количества переходов
- Использование специализированных инструкций процессора

### Профилирование
- Идентификация узких мест в коде
- Измерение времени выполнения критических секций

## Безопасность

### Защита памяти
- Проверка границ буферов
- Проверка указателей перед разыменованием
- Использование защищенных методов работы с памятью

## Процесс ревью

1. При создании или обновлении pull request автоматически запускается AI код-ревью
2. Создатель pull request должен ознакомиться с замечаниями AI и при необходимости внести исправления
3. Ревьюер должен проверить все замечания AI и убедиться в их корректности
4. Все критические замечания должны быть исправлены до мержа
5. После мержа следует провести интеграционное тестирование
